AI 小程序活动生成系统

方案思路文档

一次连调，永久复用 — 把活动开发周期从天级压缩到分钟级

1系统定位

AI 驱动的低代码活动工厂，核心能力 = 视觉识别 + 组件编排 + 参数配置自动化

核心模型

AI 生成组件代码 + 人工一次性连调 = 永久可复用的成品积木

每个组件只在入库前需要开发介入一次，之后所有活动创建都是运营自助完成。组件库像滚雪球一样越滚越大，系统越用越快。

2系统架构

两条主线 + 三层架构，组件库是连接两条主线的核心枢纽

主线 A

造积木 — 组件生产线

AI 生成组件前端代码，人工完成连调后入库，形成可直接调用的成品组件

描述需求 → AI 生成代码 → 人工连调 → 测试入库
主线 B

搭积木 — 活动生产线

运营上传 UI，AI 识别匹配组件，确认配参后发布。全程零开发介入

上传 UI → AI 识别 → 确认配参 → 预览发布
🧱 组件功能库 — 两条主线的连接枢纽，主线 A 产出成品积木，主线 B 消费积木搭建活动
LAYER 1
运营操作层

AI 会话窗口、活动创建流程、历史活动管理、预览与发布

LAYER 2
AI 引擎层

UI 视觉识别、组件匹配、布局编排、参数推理、对话式改参

LAYER 3
底座能力层

组件功能库、组件运行时、服务端接口、配置热更新发布引擎

组件数据结构
ID
标识与分类

唯一ID、组件名称、功能类别（抽奖/任务/展示/兑换）、状态（draft → testing → available）

AI
视觉特征签名

视觉描述 + 关键词，供 AI 在识别 UI 时匹配。如"圆形转盘，扇形分区，中心指针按钮"

FE
前端成品代码

AI 生成 + 人工连调后的完整组件包，含换肤配置（颜色插槽、布局模式）

P
参数 Schema

每个可配置参数的类型、默认值、是否必填。关键字段 ai_inferable：标记 AI 能否从 UI 文案推理

ai_inferable 是核心设计 —— 它决定活动创建时哪些参数 AI 自动填（如"每日3次"→ daily_limit=3），哪些必须运营手填（如奖品内容、活动时间）

4主线 B：活动生产线（核心流程）

纯运营操作，零开发介入。6 步完成从设计稿到上线活动

1

上传活动 UI 运营

在 AI 会话窗口中拖拽上传设计稿

  • 支持格式：PNG / JPG / PSD / Figma 链接
  • 支持上传多张（不同页面、弹窗、各状态截图）
  • AI 会话窗口中直接拖拽或粘贴
2

AI 识别 + 组件匹配 AI

AI 对上传的 UI 执行两个动作：区域切割 + 组件匹配

动作 1：视觉区域切割

把完整 UI 拆分为多个功能区域（顶部 Banner 区、中部抽奖区、底部任务区等）

动作 2：组件匹配

每个区域在组件库（status=available）中匹配最相似的组件

UI 区域匹配组件置信度
顶部活动 Banner96%
中部转盘抽奖91%
底部任务列表87%
悬浮层奖品记录入口90%
技术实现
  • 多模态大模型（GPT-4o / Claude）做视觉理解
  • Prompt 中注入组件库清单（名称 + 视觉特征描述）作为匹配候选
  • 输出结构化 JSON，非自由文本
3

确认组件清单 运营

AI 输出识别结果卡片，运营逐个确认或批量确认

  • ✅ 确认识别正确
  • ✏️ 修正错误匹配（从组件库中重新选择）
  • ➕ 补充遗漏的组件
  • ➖ 删除多余的识别
  • ⚠️ 库中无匹配组件时，提示"是否进入组件创建流程？"—— 无缝跳转主线 A
4

AI 自动配参 + 运营补充 AI + 运营

AI 自动编排布局、映射样式、推理参数；运营补充 AI 无法推理的参数

AI 自动执行
  • 布局编排：按 UI 空间位置排列组件渲染顺序和层级关系
  • 样式映射：提取 UI 配色 / 字体 / 圆角，写入组件样式插槽
  • 参数预填：扫描 UI 文案，推理 ai_inferable=true 的参数
🟢 AI 已推理（请确认）
  • 每日抽奖次数：3次 — 依据 UI 文案"每日3次机会"
  • 单次消耗积分：10 — 依据 UI 文案"消耗10积分"
  • 奖品格位数：8个 — 依据 UI 布局识别
🔴 需要你填写
  • 活动时间：起止日期
  • 奖品列表：名称、图标、中奖概率
  • 未中奖提示语
  • 参与条件 / 用户等级限制

运营也可以用对话方式修改："把每日次数改成5次""一等奖设成100元红包，概率1%"

5

预览 运营

真实组件渲染 + Mock 数据，所见即所得

  • PC 端即时预览活动页面
  • 支持手机扫码真机预览
  • 不满意可返回 Step 4 继续调整参数
6

发布 运营

一键发布，配置自动存入历史活动列表

  • 生成活动链接 / 小程序码
  • 配置热更新，不走小程序审核流程（配置驱动，非代码驱动）
  • 自动归档到历史活动列表，支持后续一键复制

7落地节奏

4 个阶段，约 8 周跑通核心能力，之后持续滚雪球

PHASE 1 · 3周
定标准 + 造第一批积木
  • 定义组件标准和数据结构
  • AI 生成 10 个核心组件代码
  • 人工完成后端连调入库
交付：10 个可用组件的功能库
PHASE 2 · 3周
跑通活动生产线
  • UI 识别 + 组件匹配 Prompt 调优
  • 自动配参 + 参数表单生成
  • 预览 + 发布链路
交付：端到端跑通真实活动
PHASE 3 · 2周
历史管理 + 体验打磨
  • 历史活动列表 + 一键复制
  • 会话式改参体验优化
  • 样式映射 / UI 还原度提升
交付：完整可用的产品
PHASE 4 · 持续
滚雪球
  • 每做一次新活动就积累组件
  • 组件越多 → 匹配越准
  • 创建速度持续加快
飞轮效应，越用越快

8关键风险与应对

识别核心风险，提前准备应对策略

AI 识别准确率不够
运营需大量手动修正，体验差
初期限定支持的组件类型，收窄识别范围。积累标注数据持续优化 Prompt，逐步扩展覆盖面。
UI 还原度低
发布的活动和设计稿视觉差距大
组件样式插槽足够灵活，支持深度换肤。复杂视觉效果走"UI 切图贴皮"而非代码还原。
组件库覆盖不足
遇到新玩法就卡住，无法生成
Phase 1 做好组件标准，让新组件快速入库有章可循。AI 辅助生成前端代码，压缩组件生产周期。
参数配置出错上线
线上事故，如概率配错、奖品异常
发布前强制预览 + 参数合法性校验（概率总和=100% 等）。关键参数（概率/奖品）双人确认机制。
